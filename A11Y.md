# Accessibility

## Accessibility Goals

Presto is intended to be usable through clear visual structure, keyboard-friendly interactions, properly labelled inputs, visible system feedback, and layouts that adapt without causing horizontal scrolling on smaller screens.

## Semantic Structure

Standard HTML form elements are used across authentication, presentation setup, and editor dialogs. Inputs such as text fields, password fields, textareas, selects, toggles, and buttons rely on semantic elements or accessible primitives from Radix.

Most inputs are paired with labels, covering areas like registration, login, presentation creation, text editing, image descriptions, video URLs, code input, and background settings. Where necessary, required fields are clearly indicated with visible cues.

Presentation cards are implemented as buttons rather than generic containers. This ensures they are keyboard-accessible and expose correct semantics to assistive technologies.

## Keyboard Support

Forms for login and registration can be submitted using the Enter key, following standard browser behaviour.

Users can move between slides using arrow keys in both the editor and preview modes, offering an alternative to mouse-based navigation.

Dialog interactions are handled using accessible modal patterns provided by Radix, including focus trapping and keyboard controls such as Escape to close.

To avoid unintended behaviour, slide navigation shortcuts are disabled when focus is inside text inputs or textareas, so typing does not trigger slide changes.

## Focus and Visual Feedback

Focusable elements like buttons and presentation cards display clear focus outlines. Active states in the editor, such as the selected slide or an open Slides panel, are visually distinct.

Toast notifications are used to communicate both successful actions and errors, including events like account creation, presentation setup, slide operations, and changes to element scaling or layering.

Actions with destructive outcomes are styled accordingly and typically require confirmation before proceeding.

## Colour and Contrast

The UI uses a neutral colour scheme with dark primary actions placed on lighter backgrounds for contrast. Key actions are emphasised through clear text and icon visibility, while destructive actions are highlighted in red to differentiate them.

Code blocks are displayed with a darker background and syntax highlighting, improving readability and separating them from standard content.

Navigation controls in preview mode are semi-transparent but remain large and visible enough when active. They fade out after inactivity to reduce distraction.

## Images and Media

When adding images, users are prompted to provide alt text or descriptions, encouraging better accessibility for non-visual users.

Video elements require a YouTube embed link, with autoplay exposed as an explicit option rather than being enabled by default.

## Responsive Accessibility

The layout adjusts across desktop, tablet, and mobile environments. Input fields and textareas are constrained so that long text, URLs, or code snippets do not break the layout.

On smaller screens, the Slides panel switches to a full-width view instead of remaining as a narrow sidebar. The canvas keeps a consistent 16:9 ratio and scales responsively, helping maintain an accurate preview of slide proportions.

The dashboard header remains fixed, allowing navigation to stay accessible while scrolling.

## Known Limitations

The editor currently relies heavily on pointer-based interactions for tasks like dragging, resizing, and managing layers. Providing keyboard-based alternatives for these actions would improve accessibility.

Some toolbar buttons are represented only by icons and depend on tooltips or surrounding context. Adding explicit `aria-label` attributes would make them clearer for screen reader users.

Colour selection uses the browser’s native picker, which may differ in accessibility depending on the environment.
